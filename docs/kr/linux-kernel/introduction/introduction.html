"유저랜드 pwn은 어느 정도 공부했지만, 커널은 어려워 보여서 손이 가지 않는다"라고 느끼는 분이 많을 것입니다. 하지만 실제로는 경우에 따라 커널 익스플로잇이 매우 쉬운 경우도 있습니다.
이 절에서는 유저랜드 익스플로잇과 커널 익스플로잇의 차이, 환경 구성 방법 등을 설명합니다.

커널 익스플로잇의 특징

먼저 유저랜드와 비교했을 때 커널 공간의 취약점이 어떤 특징을 가지는지 알아봅시다.

공격 대상

유저랜드 익스플로잇과 커널 익스플로잇의 가장 큰 차이는 목표에 있습니다.
지금까지 설명한 유저랜드 익스플로잇에서는 대부분 "임의 명령 실행"을 목표로 익스플로잇을 작성했습니다. 반면 커널 익스플로잇은 일반적으로 "권한 상승"을 목표로 작성합니다. 공격자가 어떤 수단으로든 대상 머신에 침입한 상태를 가정하고, 커널 익스플로잇을 이용해 추가로 root 권한을 획득하는 것입니다[1]. 이렇게 로컬 머신에서 권한을 올리는 것을 LPE(Local Privilege Escalation)라고 합니다.
물론 유저랜드에서도 권한 상승이 가능한 취약점이 있을 수 있지만, 그것은 공격 대상 프로그램이 특권 사용자로 실행되기 때문입니다. 커널 익스플로잇의 경우 공격 대상은 주로 다음 두 가지입니다.

  1. Linux 커널
  2. 커널 모듈

Linux 커널 내부의 코드(시스템 콜, 파일 시스템 등)는 root 권한으로 동작하므로, 커널 자체에 버그가 있으면 LPE로 이어질 수 있습니다.
다른 하나는 디바이스 드라이버 같은 커널 모듈에 포함된 취약점입니다. 디바이스 드라이버는 사용자 공간에서 외부 장치(프린터 등)를 보다 쉽게 다루기 위한 인터페이스입니다. 디바이스 드라이버 역시 항상 root 권한으로 동작하므로[2], 버그가 있으면 LPE로 이어질 수 있습니다.

공격 방식

유저랜드 익스플로잇에서는 보통 공격 대상 서비스에 입력을 주는 방식으로 익스플로잇했습니다. 그래서 Python 같은 언어로 익스플로잇을 작성하는 경우가 많습니다.
반면 커널 익스플로잇에서 대상은 OS나 드라이버입니다. 이런 조작은 훨씬 저수준이기 때문에 보통 C 언어로 익스플로잇을 작성합니다. 물론 Python 등으로도 쓸 수는 있지만, 공격 대상 머신 특히 CTF나 실습 환경처럼 작은 Linux 환경에는 Python이 없는 경우가 많아서 실행되지 않을 가능성이 큽니다.

이 사이트에서도 익스플로잇은 C 언어로 작성합니다. 자세한 이야기는 다른 절에서 다시 나오지만, musl-gcc라는 컴파일러를 사용합니다.

자원의 공유

커널 익스플로잇의 또 다른 특징은 자원이 공유된다는 점입니다.
유저랜드에서는 보통 공격 대상 프로세스가 하나 존재하고, 그 프로세스를 익스플로잇해서 셸을 따는 식으로 공격했습니다. 반면 Linux 커널이나 디바이스 드라이버 같은 프로그램은 OS를 사용하는 모든 프로세스가 공유합니다. 시스템 콜은 누구나 원하는 시점에 호출할 수 있고, 디바이스 드라이버도 누가 언제 접근할지 알 수 없습니다. 즉 커널 공간에서 동작하는 코드를 작성할 때는 항상 멀티스레드 환경을 염두에 두고 프로그래밍하지 않으면 쉽게 취약점을 심게 됩니다.

전역 변수처럼 경합이 생길 수 있는 데이터를 사용할 때는 락을 잡아야 한다는 뜻이네.
커널 공간 프로그래밍은 정말 어렵다.

힙 영역의 공유

게다가 커널의 힙 영역은 모든 드라이버와 커널이 공유한다는 큰 특징이 있습니다.
지금까지 유저랜드 익스플로잇에서는 프로그램마다 힙이 따로 있었기 때문에, 어떤 프로그램에서 Heap Overflow가 발생했다고 해서 그것이 exploitable한지는 그 프로그램의 구조에 달려 있었습니다. 하지만 예를 들어 디바이스 드라이버에서 한 번 힙 오버플로가 발생하면, 다른 디바이스 드라이버나 Linux 커널이 힙에 확보한 인접 데이터까지 오염시킬 수 있습니다.

공격자 관점에서 이 특징은 장점과 단점을 모두 가집니다. 장점은 힙 주변의 작은 취약점도 LPE로 이어질 가능성이 매우 높다는 점입니다. 예를 들어 Linux 커널에는 함수 포인터를 가진 객체가 많이 존재하므로, 그중 하나를 활용해 RIP를 쉽게 장악할 수 있습니다. 단점은 전체 프로그램의 영향을 받기 때문에 힙 상태를 예측하기 어렵다는 점입니다. 유저랜드 프로그램은 단순한 경우 입력에 대해 힙 상태가 결정적이었기 때문에, 이른바 House of XXX 같은 복잡한 힙 익스플로잇도 가능했습니다. 반면 커널에서는 Heap Overflow가 발생한 청크 뒤에 어떤 데이터가 있는지, Use-after-Free로 free된 뒤 누가 그 주소를 다시 사용할지 알 수 없습니다.

그렇다면 커널 익스플로잇에서는 Heap Spray가 중요하겠네.

취약점 자체는 유저랜드와 크게 다르지 않습니다. Stack Overflow나 Use-after-Free 같은 버그는 커널에도 존재할 수 있습니다. 또한 디바이스 드라이버의 스택에도 보안 기법으로 Stack Canary를 둘 수 있습니다. 다만 커널 공간 특유의 취약점도 있으므로, 그것은 뒤 절에서 등장합니다.

qemu의 사용

Linux 커널 익스플로잇을 작성할 때는 디버깅을 위해 에뮬레이터 위에서 커널을 실행합니다. VM은 무엇이든 괜찮지만, 일반적으로 qemu를 많이 쓰므로 이 사이트에서도 qemu를 사용합니다.

디스크 이미지

qemu로 머신을 부팅할 때는 Linux 커널과 별도로 루트 디렉터리로 마운트될 디스크 이미지가 필요합니다.
디스크 이미지는 일반적으로 ext 같은 파일 시스템의 raw 바이너리이거나, cpio라고 불리는 형식으로 만들어져 배포됩니다.
